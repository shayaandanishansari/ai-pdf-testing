# Arabic PDF Text Extraction

A research project for extracting and summarizing text from Arabic PDFs. The project benchmarks several approaches — from local Python libraries to large AI vision models — to find what actually produces readable Arabic output.

---

## The Problem

Arabic PDFs are notoriously difficult to extract text from. Two specific issues affect nearly all standard extraction libraries:

1. **Visual order encoding** — Arabic is written right-to-left, but many PDF generators store glyphs in left-to-right visual order. Libraries that read the raw character stream return words with their letters reversed.
2. **Detached diacritics** — Vowel marks (harakat like fatha, kasra, damma) are stored as separate positioned glyphs rather than as Unicode combining characters attached to their base letter. This scatters them across the extracted text.

The result is that `pdfplumber` and basic `PyMuPDF` both produce garbled Arabic even though the PDF visually renders correctly.

## Approaches

### Local library extractions

| Script | Approach | Output file |
|---|---|---|
| `extract_pdfplumber.py` | `pdfplumber` — baseline, character-per-line output for Arabic | `arabic-book-sample-pdfplumber.txt` |
| `extract_pymupdf.py` | `PyMuPDF` basic `get_text()` — words on single lines but letters reversed | `arabic-book-sample-pymupdf.txt` |
| `extract_pymupdf_coords.py` | `PyMuPDF` rawdict + coordinate-based RTL sort + diacritic bubble fix | `arabic-book-sample-pymupdf-coords.txt` |

`extract_pymupdf_coords.py` is the most sophisticated local approach. It:
- Pulls every character's exact X coordinate from the PDF via `get_text("rawdict")`
- Sorts characters right-to-left when a line is predominantly Arabic (>30% Arabic Unicode) or when PyMuPDF's own direction flag signals RTL
- Bubbles detached combining diacritics to follow their base character

### AI API extractions

| Script | Model | How the PDF is sent | Output file |
|---|---|---|---|
| `extract_gemini.py` | `gemini-2.0-flash` | Entire PDF as inline base64 (`application/pdf`) | `arabic-book-sample-gemini.txt` |
| `extract_qwen.py` | `qwen3.5-397b-a17b` | Each page rendered to 2× PNG, sent as `image_url` base64 | `arabic-book-sample-qwen.txt` |

### AI API summarization

| Script | Model | Output file |
|---|---|---|
| `summarize_gemini.py` | `gemini-2.0-flash` | `arabic-book-sample-gemini-summary.txt` |
| `summarize_qwen.py` | `qwen3.5-397b-a17b` | `arabic-book-sample-qwen-summary.txt` |

## Setup

### Prerequisites

- Python 3.10+

### Install dependencies

```bash
pip install pdfplumber pymupdf httpx python-dotenv
```

### API keys

Create a `.env` file in the project root:

```
CLAUDE=your_claude_api_key
MISTRAL=your_mistral_api_key
QWEN_API_KEY=your_dashscope_api_key
GEMINI_API_KEY=your_gemini_api_key
```

- **Gemini** key: [Google AI Studio](https://aistudio.google.com/)
- **Qwen** key: [Alibaba Cloud Model Studio](https://dashscope.aliyuncs.com/) — use the international account for `dashscope-intl.aliyuncs.com`

---

## Usage

Run any script from the project root. Each script reads `arabic-book-sample.pdf` and writes its output file to the same directory.

```bash
# Local extractions (no API key needed)
python extract_pdfplumber.py
python extract_pymupdf.py
python extract_pymupdf_coords.py

# AI extractions (requires API keys in .env)
python extract_gemini.py
python extract_qwen.py

# AI summarization (requires API keys in .env)
python summarize_gemini.py
python summarize_qwen.py
```

## Notes

- The Gemini scripts send the PDF as a single base64-encoded blob. Gemini natively understands `application/pdf` so no page rendering is needed.
- The Qwen scripts render each page to a 2× resolution PNG using `PyMuPDF` before encoding, since the Qwen vision API accepts `image_url` content items rather than raw PDF bytes.
- All API calls are made with `httpx` directly — no SDK wrappers.
- `.env` is gitignored. Never commit API keys.
